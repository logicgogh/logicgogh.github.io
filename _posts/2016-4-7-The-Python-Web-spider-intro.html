Python爬虫入门及实践¶

在上篇我们介绍了Python的入门及一些基本应用, 但是作为一门脚本语言, Python的流行很大程度上是因其在网络方面的应用. 特别是在以Django为代表的一大批网络框架的兴起, Python成为网络圈的新宠(现在可能让位给Nodejs了). 我们今天要讲的网络爬虫, 也属于网络应用的一个分支.

  • 什么是网络爬虫

互联网是一张巨大的网, 网络上成千上万的网页是这张网的一个个节点, 网页上的一个个超链接是连接这些节点的线. 在这张网上, Python爬虫就像一只蜘蛛, 爬过网上的每一个节点, 把想要的信息储存下来.

  • 爬虫的基本原理

想要写出来一个Python爬虫, 我们先提纲挈领的把握以下爬虫要实现的功能和原理, 用最简单的情况实现一个例子, 往后我们就细节提升再具体讨论.

  1. 下载网页

爬虫本质上是模仿人类浏览网页, 并将人们感兴趣的信息保留下来. 既然如何我们就需要先把需要的网页下载下来. 互联网上的网页成千上万, 我们需要根据我们的需求适当的选区一个合理的开始页面.

这里举一个非常简单的例子, 我们假设需要把物理学院主页上的统计爬去下来, 保留相关的信息. 在这个例子中,我们不使用第三方现成的爬虫库,而仅仅使用Python自带的库实现相关功能.

通过浏览网页并分析, 我们发现了可以将通知页面的头一页(http://physics.nankai.edu.cn/572/list.htm) 当做起始页.

下面我们将这个网页下载下来:

In [1]:
import urllib.request
start_page = 'http://physics.nankai.edu.cn/572/list.htm'
with urllib.request.urlopen(start_page) as page:
    html = page.read()
print(html)

所以人们发明了面向对象的做法, 也就是使用类: 使用类来把变量和方法包装成一个想要的数据结构, 每次处理数据就生成一个相关的实例. 这些实例之间没有依赖关系, 更变于内存管理, 也给日后的大规模并行运算提供了基础条件.

class Spyder:
    def __init__(self, name , url, todols):
        self.name = name
        self.url = url
        self.get_page()
        self.get_notifies()
        self.get_nextPage(todols)
        
    def get_page(self):
        with urllib.request.urlopen(self.url) as page:
            html = page.read()
            self.html = html.decode('utf-8')         
    
    def get_notifies(self):
        p = re.compile("<a\shref='/\d\d.+?\starget.+?\stitle.+?\salt.+?</a>")
        pl = p.findall(self.html)
        self.items = pl      
    
    def get_items(self):
        return self.items   
    
    def get_nextPage(self, todols):
        nbp = re.compile('<a\s.+?href="(.+?)".+?下一页')
        npurl = nbp.findall(self.html)
        todols.append(npurl[0])

建议和改进¶

下面我们来回顾分析一下, 我们刚刚写的爬虫的不足之处以及改进方案.

  • 普适性

我们这个爬虫, 是一个专用爬虫,也就是爬取的师固定的网页, 经过我们分析, 很容易找出规律, 这样直接 把规律写代码(硬编码), 做出来的爬虫只能适用与一个网站.

我们可以把不同网站都要实现的功能抽象出来形成一个抽象的父类. 把每一个不同网页的规则作为子类的方法, 这样可以做到普遍实用性.

  • 逻辑性

在上述例子中,有一个很不优雅的设计, 就是todol这个记录下一页的数组. 首先师普适性, 有的网页不是单项链接的, 可能从这个网页上能有许许多多的超链接. 其次, 我们应该把每一个处理完的网页链接从表中移除,移到另一个代表着已经完成的列表里.

这时,Python的数组类型显得有些笨拙. 我们期待这一种类似与堆栈的数据结构, 能让我们从这个栈中弹出一个链接给爬虫处理,并把这个链接压倒另一个代表已完成的栈里.

值得庆幸的是, Python中的内置Queue库, 提供了这样的非常易用且高效的列队堆栈结构. 我们可以直接用它替换掉例子中的todol.

  • 迎来大数据

所有的问题, 在数据量小的时候, 都不是问题

但是在数据量特别庞大的, 评估程序复杂度和合理的资源分配问题就变得很突出.

1) 判断重复

就拿一个很简单的例子, 我们每进行一次爬取动作, 应该先该在已爬取的列表中查看有没有爬取过了. 那么在一个hash表中进行查询的复杂度是表中元素数N的对数:log(N), 那么对于N个网址来说, 就是N*log(N) 的复杂度.

小数据量无关紧要, 但是试想, 如果像谷歌这样的大网站呢?(谷歌实际上就是把全世界所有的网页都爬取下来, 然后用它发明的RankPage算法做排序. ) 这个网页的数据量会把内存和CPU都撑爆. 也许我们不用爬取全世界所有的网页, 但是就单单对一个商业网站的爬取(比如说weibo), 数据量就大的惊人. 

通常的判重做法是怎样呢？<span style="color:red">Bloom Filter</span>. 简单讲它仍然是一种hash的方法，但是它的特点是，它可以使用固定的内存（不随url的数量而增长）以O(1)的效率判定url是否已经在set中。可惜天下没有白吃的午餐，它的唯一问题在于，如果这个url不在set中，BF可以100%确定这个url没有看过。但是如果这个url在set中，它会告诉你：这个url应该已经出现过，不过我有2%的不确定性。注意这里的不确定性在你分配的内存足够大的时候，可以变得很小很少。

这是一种近似算法.有兴趣的可以去参考相关资料.

2) 集群化

如果我们要做的是大规模的数据抓取和分析储存的话, 那么单单用一台电脑的性能, 肯定是需要话费大量的时间. 

那么就不得不使用集群. 

牵扯到集群, 就必须要引入一堆相关的集群和并行运算的技术. 谈论到这里, 我们就发现了我们在上文中把抓取过程重构成为一个类的意义了: 便于分布式计算.

这里我们把上述流程再优化一下:

假设我们有一个集群,里面有上百个服务器可以帮我们运行任务.

那么我们首先在一个master节点上(主服务区), 部署以下我们的todol列表, 已爬取网址列表, 以及判断是否爬取过的Bloom Filter方法. 

接着我们在master上, 从todol中分配出一个个网址, 这些网址被Spyder类实例化后, 分布在每一台slave节点(侍从服务器)上运算, 然后运算结果返回到主服务器定义的储存单元上, 再把需要抓取的网址, 存到主服务器的todol列表里. 循环往复.

还是那句话,牵扯到大数据量, 所有的小问题都是大问题. 

我们来分析以下, 上文提到的过程中出现的问题和解决方案:

首先我们需要有一个可靠的堆栈结构, 是的它可以同时被很多服务区及其以上的多个进程同时访问和读写. 这个功能是一个简单的数组或者queue不能完成的. 我们需要借助分布式存取的数据库(如redis).

另外, 大量的数据的涌入, 还需要很大的个能都可伸缩的告诉的储存系统. 这个分布式文件系统, 我们可以采用Hadoop, 没错, 它就是谷歌用来储存海量数据的文件系统的开源版本. 

最后还有许多细小的地方比如容错性和可扩展性等.
  • 数据挖掘

当我们如愿的把所有的数据都按照要求抓取下来并储存完好的时候, 剩下的就是如何再这些数据中, 找出我们需要的数据, 进行有效的处理.

这个可以和上次讲的pandas的数据分析工具结合起来使用, 还会用到许许多多的数据统计方法,这里就不展开讲了.

现成方案¶

虽然在写爬虫的时候有许许多多需要考虑的问题, 但是很多人都已经考虑过了并且给出了不错的框架可用.

抛开大数据和集群不谈, 只谈我们上文实现的mini版的爬虫来讲, 现成的框架有:

  • 网页解析: beautiful soup

  • 通用爬虫框架: Scrapy

使用他们的组合,即可解决我们大多数爬虫需求.
